1、MySQL查询优化进一步学习

1.1、子查询优化

MySQL支持的子查询优化策略如下：

使用MySQL trace工具，跟踪优化器对SQL的整个优化执行过程。

首先建三张表，并添加索引

DROP TABLE IF EXISTS `t1`;
CREATE TABLE `t1`  (
  `a1` int(11) NOT NULL,
  `b1` int(11) NOT NULL,
  INDEX `idx_a1`(`a1`) USING BTREE
) ENGINE = InnoDB CHARACTER SET = utf8 COLLATE = utf8_general_ci ROW_FORMAT = Dynamic;

DROP TABLE IF EXISTS `t2`;
CREATE TABLE `t2`  (
  `a2` int(11) NOT NULL,
  `b2` int(11) NOT NULL,
  INDEX `idx_a2`(`a2`) USING BTREE
) ENGINE = InnoDB CHARACTER SET = utf8 COLLATE = utf8_general_ci ROW_FORMAT = Dynamic;

DROP TABLE IF EXISTS `t3`;
CREATE TABLE `t3`  (
  `a3` int(11) NOT NULL,
  `b3` int(11) NOT NULL,
  INDEX `idx_a3`(`a3`) USING BTREE
) ENGINE = InnoDB CHARACTER SET = utf8 COLLATE = utf8_general_ci ROW_FORMAT = Dynamic;

然后每张表插入1万条数据

CREATE DEFINER=`root`@`localhost` PROCEDURE `proc_insert_data`()
BEGIN
        declare i int default 0;
        set autocommit = 0;
        while i < 10000
        do
            insert into t3(a3,b3) values(i+1, i+2);
        set i=i+1;
        if
            i%2000=0
        then
            commit;
        end if;
        end while;
END

然后执行下面查询语句：

select * from t1 where t1.a1 in (select a2 from t2 where t2.a2 > 30);

查询执行计划如下：

配置trace参数：

set OPTIMIZER_TRACE="enabled=on",END_MARKERS_IN_JSON=on,optimizer_trace_max_mem_size=100000;

然后执行下面语句查看最新一条SQL的查询计划生成过程：

select * from information_schema.optimizer_trace\G;

部分截图如下：

分析trace记录，经查询优化器处理后的SQL如下：

select `t1`.`a1` AS `a1`,`t1`.`b1` AS `b1` 
from `t1` semi join (`t2`) 
where (``.`a2` = `t1`.`a1`) and (`t2`.`a2` > 30)

结合查询计划可知t2表被物化之后被上拉到表t1层与t1执行了半连接，即MySQL支持将IN非相关子查询转换为半连接。

1.3、消除外连接

当内表符合“空值拒绝”条件时，外连接可以转换为内连接。

首先建立三张表，并插入数据

create table t_1(t_1_id int unique, t_1_col_1 int, t_1_col_2 varchar(10));
create table t_2(t_2_id int unique, t_2_col_1 int, t_2_col_2 varchar(10));

insert into t_1 values(1,11,'t_1_1');
insert into t_1 values(2,12,NULL);
insert into t_1 values(3,NULL,'t_1_3');
insert into t_1 values(4,14,'t_1_4');
insert into t_1 values(5,15,NULL);
insert into t_1 values(7,NULL,NULL);

insert into t_2 values(1,11,'t_2_1');
insert into t_2 values(2,NULL,'t_2_2');
insert into t_2 values(3,13,NULL);
insert into t_2 values(4,14,'t_2_4');
insert into t_2 values(6,16,'t_2_6');
insert into t_2 values(7,NULL,NULL);

对比下面三条SQL和相应查询计划：

select * from t_1 left join t_2 on true where t_1_id=t_2_id;
select * from t_1 left join t_2 on t_1_id=t_2_id;
select * from t_1 left join t_2 on t_1_id=t_2_id where t_1_id=t_2_id;

explain extended select * from t_1 left join t_2 on true where t_1_id=t_2_id;
explain extended select * from t_1 left join t_2 on t_1_id=t_2_id;
explain extended select * from t_1 left join t_2 on t_1_id=t_2_id where t_1_id=t_2_id;

对于查询1和3，结果是一样的，如下：

因为语句1和3都使用了where t_1_id=t_2_id，满足空值拒绝的条件，因此可以转换为内连接。

此外，他们的查询计划也是相同的：

查看trace可知优化后的查询语句如下：

select sql_no_cache `t_1`.`t_1_id` AS `t_1_id`,`t_1`.`t_1_col_1` AS `t_1_col_1`,`t_1`.`t_1_col_2` AS `t_1_col_2`,`t_2`.`t_2_id` AS `t_2_id`,`t_2`.`t_2_col_1` AS `t_2_col_1`,`t_2`.`t_2_col_2` AS `t_2_col_2` 
from `t_1` join `t_2` 
where (`t_1`.`t_1_id` = `t_2`.`t_2_id`)

确实转成了内连接。

而对于第2条语句，因为没有where条件，只有join/on，对于t_id_1=5时，右表只有NULL进行匹配，并且不会拒绝。因此不满足空值拒绝的条件，从而不能转换为内连接。执行结果如下：

查询计划如下：

与上面唯一不同的是第一行的possible_keys为NULL

优化后的查询语句如下：

select sql_no_cache `t_1`.`t_1_id` AS `t_1_id`,`t_1`.`t_1_col_1` AS `t_1_col_1`,`t_1`.`t_1_col_2` AS `t_1_col_2`,`t_2`.`t_2_id` AS `t_2_id`,`t_2`.`t_2_col_1` AS `t_2_col_1`,`t_2`.`t_2_col_2` AS `t_2_col_2` 
from `t_1` left join `t_2` 
on(`t_1`.`t_1_id` = `t_2`.`t_2_id`)

可以看到仍然是left join并没有转为内连接

1.4、语义优化

1.4.1、检测空回答集

首先，建表插入数据

create table student(name varchar(30) not null, age int);
insert into student values('antirez',23);
insert into student values('James Gosling',22);
insert into student values('Rod Johnson',19);

对于下面查询语句：

select * from student where name is NULL and age > 18

查看查询计划：

优化后的SQL如下：

select `student`.`name` AS `name`,`student`.`age` AS `age` 
from `student` 
where 0

从查询计划上看，在列name上进行了过滤操作，因此extra是impossible where， 由此可知MySQL是可以对空值约束进行语义优化的

1.4.2、利用索引优化排序操作

建表

create table t4(a int, b int unique);

首先看下面两条SQL：

explain extended select * from t4 tmp1, t4 tmp2 where tmp1.b = tmp2.b order by tmp1.b 
explain extended select * from t4 tmp1, t4 tmp2 where tmp1.a = tmp2.a order by tmp1.a

优化后的SQL分别如下：

select `tmp1`.`a` AS `a`,`tmp1`.`b` AS `b`,`tmp2`.`a` AS `a`,`tmp2`.`b` AS `b` 
from `t4` `tmp1` join `t4` `tmp2` 
where (`tmp1`.`b` = `tmp2`.`b`) 
order by `tmp1`.`b`

select `tmp1`.`a` AS `a`,`tmp1`.`b` AS `b`,`tmp2`.`a` AS `a`,`tmp2`.`b` AS `b` 
from `t4` `tmp1` join `t4` `tmp2` 
where (`tmp1`.`a` = `tmp2`.`a`) 
order by `tmp1`.`a`

从查询计划和优化后的SQL来看，对第一条语句，仍然是先连接后排序，MySQL对此没有进行优化；对第二条语句，连接之后用临时文件进行了排序操作，也没有进行优化。

再看下面两条SQL，只在单表的索引列上进行排序查询：

explain extended select * from t4 order by b;
explain extended select b from t4 order by b;

从查询计划看，对第一条SQL，type为ALL，即进行了全表扫描，extra为using filesort表示进行了排序，没有利用索引进行排序优化；对第二条SQL，type为index，即进行了索引扫描，extra为using index，表示利用索引进行了排序优化。

分析以上实验可知：MySQL支持利用索引优化排序操作，但是条件很严格（选择列只有排序列，且为索引列）。

1.5、非SPJ优化

1.5.1、group by优化

常规处理方法是创建一个临时表执行分组操作，对他的优化一般是利用索引，有loose index scan和tight index scan两种方式

建表：

create table t_god(a int, b int, c int, d int, e int);
create index t_god_idx_1 on t_god(a,b,c);
create index t_god_idx_2 on t_god(d);

执行下面SQL：

explain extended select a from t_god group by a;

type为index表示进行了索引扫描，extra为using index表示利用索引进行了分组优化。

explain extended select a,min(b) from t_god where c>2 group by a;

从查询计划看，使用索引对带有min聚合函数的分组进行了优化

1.5.2、order by优化

MySQL提供了两种排序算法，分别是双路排序算法和单路排序算法

  • 双路排序算法：只对order by子句包含的列进行排序，适用于blob和text类型列参与的排序
  • 单路排序算法：除了order by子句包含的列，还利用查询目标列进行排序，适用于除blob和text类型外的所有类型的排序

建表：

create table t_o1(a1 int unique, b1 int);

看下面两条SQL：

explain extended select * from t_o1 order by a1
explain extended select a1 from t_o1 order by a1

由查询计划可知，第一条SQL使用了全表扫描，并进行了排序（using filesort），但没有使用索引进行优化；第二条SQL使用索引扫描，且用索引对排序进行了优化。对比可知，MySQL支持用索引消除排序技术，但仅限于目标列是索引列。

1.5.3、distinct优化

建表

create table t5(a int primary key, b int);

执行下面SQL：

explain extended select distinct a from t5;

从查询计划看，在表t5上执行了索引扫描（type=index），并且利用索引（extra=using index）对distinct进行了优化

1.5.4、limit优化

MySQL支持对不带having语句的limit进行优化，主要优化点如下：

  • 对单表扫描的影响：索引扫描代价更小，使用索引扫描优化
  • 对排序的影响：如果limit和order by一起使用，当取到limit设定的个数后，停止排序
  • 对去重的影响：如果limit和distinct一起使用，当取到limit设定的个数后，停止去重操作
  • 受分组的影响：如果limit和group by一起使用，group by按索引有序计算每个组的总数过程中，limit操作不必计数直到下一个分组开始计算
  • Limit 0：直接返回空结果集

单表排序 + limit操作的格式如下：

select ... from single_table ... order by non_index_column [desc] limit [M,]N;

limit对排序算法的影响如下：

  • 通常排序算法：
    • 扫描表，重复以下步骤直到文件结束
      • 把扫描的元组放入排序的缓存直到缓存填满
      • 把排序缓存中的前N（或者M+N）行写入一个“待归并文件”
    • 对归并文件排序
    • 按序返回所有文件的前N行（指定M的话，就跳过M行再取N行）
  • 受limit操作影响改进的算法：（适用于N（M+N）行待排序的元素小于排序缓存）
    • 扫描表，把每一个满足条件的元组插入一个按排序有序的队列
    • 如果队列满，则淘汰最后一个再插入新的
    • 返回前N行（如果指定M，则跳过M行再取N行）

2、和张柯学长的讨论

2.1、多表连接查询优化部分的代价估算怎么做？如何改？

  • 多表连接里是否下推

  • 从MySQL出发，针对不同版本ma的特性，优化器代价估算应该做哪些调整

    • 分离版：写变快，读变慢，应该减少写代价，增加读代价，并且考虑网络代价和磁盘IO的关系（常数）
    • 主从版：对于写操作，CE节点CPU代价变大；SE节点日志应用代价（同步）
    • 多SE版：写变慢，raft比Quorum快，读目前和分离版一样

2.2、查询重写-查询计划树下沉 和 SQL语句改写下沉，哪个方案更好？实现有什么问题

下推查询树 下推SQL
实现复杂性 非常复杂，工作量大 相对容易实现
网络传输代价 很大 较小（信息传输密度大）
CE计算代价 序列化代价 从查询计划树转换成SQL的代价
SE计算代价 反序列化代价 解析SQL生成查询计划树
可移植性 较差 较好
优势 表比较多 表较少

2.3. 能不能做SE层带数据版本的查询？（把CE事务可见 对应为：SE数据版本 是否可行？）

理论上可行。这个要参考MVCC和HTAP相关论文，进一步学习调研。

可能实现思路：

  • 延缓和OLAP请求的页相关的redo日志的应用（可能产生redo堆积）
  • 为OLTP需要应用redo的页保留副本（可能产生较多副本，占用较大内存）
